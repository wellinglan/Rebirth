# Rebirth v0.7.0-alpha Release Notes

## Release Positioning

This is an Alpha acceptance build for personal evaluation. It is not a production-stable release, and no GitHub Release or binary artifact is published by this document.

## Included Capabilities

- **Today**: daily priorities, mood, energy, time, note, health summary, history, and local persistence.
- **Journal**: structured daily reflection with local history.
- **Plan**: local goal hierarchy and status management.
- **Health**: local sleep, exercise, body-state, and related health records.
- **Growth**: local 7-day and 30-day trends based on existing records.
- **Profile Sync**: explicit development login, device registration, and manual canonical Profile sync.
- **AI Consent**: generation is blocked until the user explicitly enables consent and chooses data scopes.
- **AI Request Preview**: minimized input is assembled locally and shown before final confirmation.
- **Weekly Report**: one explicit, synchronous, non-streaming generation flow.
- **AI History**: local pending, completed, failed, and soft-deleted report states.
- **Pending Recovery**: status-only reconciliation through the original endpoint and cloud account binding.

## Platforms And Providers

Windows and Android are build targets. Windows build/start and automated full-stack coverage are available; the physical Android matrix must remain `NOT EXECUTED` until a real device is connected and tested.

The default Server Provider is `disabled`. The deterministic `fake` Provider is for development and acceptance testing and never calls OpenAI. The `openai` Provider requires a Server-only API key and may incur cost. A real OpenAI smoke is optional and is not part of normal CI.

## Data And Retention

Core records are local-first in Flutter SQLite. Profile sync is manual and currently covers only the documented Profile contract. AI input payloads, canonical JSON, sources, notes, and Provider request/response bodies are not stored in the Server ledger. Validated AI output is retained for the configured result TTL; minimal dedupe tombstones remain for the longer dedupe TTL. Cleanup is lazy on request entry and can also be invoked through the maintenance CLI.

Soft-deleting a local AI report does not delete source Today, Journal, Health, Plan, or Growth data and does not erase the independent Server dedupe ledger.

## Known Limitations

- AI ownership is durable and at-most-once for a retained user/request ID, but the system is not exactly-once.
- A Provider-return/database-commit crash gap can become `outcome_unknown`; Generate POST is never automatically retried.
- OpenAI `store=false` is not an absolute zero-retention guarantee.
- Session and request-binding metadata use SharedPreferences and are development-level storage, not secure production credential storage.
- Production still requires HTTPS, managed secrets, backup/restore procedures, database encryption/access controls, monitoring, rate limiting, legal/privacy review, and a deletion policy.
- Android still uses the development application ID `com.example.rebirth` and debug signing for release-mode builds. The generated APKs are for Alpha acceptance only, not store or production distribution.
- Profile sync does not include Today, Journal, Plan, Health, Growth, or AIReport records.
- Android physical-device acceptance and the optional paid OpenAI smoke are recorded separately and must not be inferred from successful builds.

## Backup Reminder

Back up the local Flutter database and Server database before installing a new Alpha build or applying Server migrations. Keep backups outside the active database directory and verify restore steps before relying on the build for long-term personal data.
